Below is a **practical architecture and plan** to build a **comprehensive agent-facing knowledge system** that the research agent can query during campaigns. The goal is to expose financial knowledge, event semantics, detector behavior, and historical reflections in a way that is **discoverable, structured, and safe for autonomous decision loops**.

The design assumes:

* the **platform remains the execution engine**
* the **agent queries knowledge and memory**
* the **knowledge system improves search quality**
* all information is **machine-readable and deterministic**

---

# 1. Goal of the Knowledge System

Provide the agent with a **discoverable research knowledge base** that contains:

1. **financial event semantics**
2. **detector mechanics**
3. **market microstructure theory**
4. **trigger usage guidance**
5. **historical campaign reflections**
6. **strategy heuristics**
7. **market regime knowledge**

This allows the agent to reason with **domain knowledge + empirical campaign results**.

---

# 2. Core Design Principles

### Principle 1 — Platform remains authoritative

Registries remain the **source of truth**.

Knowledge documents only provide **interpretation and context**.

---

### Principle 2 — Structured discoverability

Documents must be organized so the agent can find them by:

* event name
* detector name
* event family
* market concept
* trigger type
* context

---

### Principle 3 — Agent-readable

Documents must be:

* concise
* structured
* queryable
* consistent schema

---

### Principle 4 — Reflection integration

Campaign reflections become part of the knowledge system.

---

### Principle 5 — No speculative alpha

Documents should describe **market mechanisms**, not guaranteed signals.

* a code-editing agent that mutates detectors/registries during campaigns
* a free-form strategist that writes arbitrary configs without platform mediation
* an in-process “AI stage” buried inside the pipeline
* a stateless experiment generator
* a planner that directly interprets raw ledger tables without summarized campaign state

All of those designs weaken reproducibility, search discipline, or debuggability.

# Final complete agent design

Use a **four-layer architecture**:

```text
Layer 1: Platform Core
Layer 2: Campaign Intelligence + Knowledge
Layer 3: External Research Agent
Layer 4: Campaign Controller
```

Each layer has a precise role.

---

# Layer 1 — Platform Core

This should remain deterministic and non-agentic.

## Responsibilities

* registries as source of truth
* config loading
* validation
* trigger expansion
* dependency resolution
* execution
* evaluation
* provenance artifacts
* tested ledger
* support matrix enforcement

## It should own

* legality
* semantics
* execution
* auditability

## It should never own

* heuristic search policy
* free-form reasoning
* narrative research strategy

This part is already largely done.

---

# Layer 2 — Campaign Intelligence and Knowledge

This is the interface between raw platform outputs and the agent.

It should expose two kinds of information:

## A. Dynamic campaign intelligence

Derived from ledger and run artifacts.

Artifacts:

* `campaign_summary.json`
* `search_frontier.json`
* `event_statistics.json`
* `campaign_findings.json`
* `reflections.json`

## B. Static and semi-static knowledge

Curated knowledge for semantic reasoning.

Artifacts:

* event docs
* detector docs
* template docs
* market mechanism docs
* ontology / graph knowledge
* support matrix
* search heuristics

This layer should answer:

* what has worked?
* what has failed?
* what remains unexplored?
* what is this event/mechanism/template actually about?
* what adjacent search regions are sensible?

This is where the knowledge graph belongs.

---

# Layer 3 — External Research Agent

This is the LLM-driven decision maker.

## Core role

The agent should do exactly five things:

1. read campaign intelligence
2. read relevant knowledge/ontology context
3. choose one bounded search intent
4. emit one deterministic experiment proposal
5. append one structured reflection after the run

That is all.

## The agent should not

* edit code
* edit registries
* directly write ledger rows
* reinterpret platform legality
* bypass the support matrix
* directly manipulate detector execution

## The agent’s job is not “be smart in general”

Its job is:

**choose the next best legal experiment under evidence and budget constraints**

That is a much narrower and better-defined role.

---

# Layer 4 — Campaign Controller

This should be a thin orchestrator, not an intelligence layer.

## Responsibilities

* load current campaign state
* query intelligence artifacts
* optionally query knowledge graph
* call the LLM with a strict prompt contract
* receive a proposed config
* validate config through platform
* execute run
* refresh intelligence artifacts
* append reflection
* enforce campaign-level state transitions

This is the piece that turns the platform plus agent into a real loop.

---

# Final operating loop

The correct final loop is:

```text
1. campaign controller reads campaign state
2. campaign intelligence artifacts are refreshed
3. controller queries knowledge graph for relevant context
4. external agent receives:
   - campaign_summary
   - search_frontier
   - relevant ontology/knowledge snippets
   - support matrix
   - hard limits
   - prior reflection memory
5. agent emits:
   - one deterministic experiment_config
   - one declared intent
   - one next-step decision rule
6. platform validates or rejects
7. platform executes
8. platform writes:
   - provenance artifacts
   - results
   - ledger updates
9. controller synthesizes reflection
10. reflection appended to campaign memory
11. campaign intelligence refreshed
12. loop repeats or halts
```

That is the final correct design.

# Final agent internal design

The agent itself should have five explicit internal modules, even if implemented within one prompt contract.

## 1. Situation Assessment

Inputs:

* `campaign_summary.json`
* `search_frontier.json`
* recent reflections
* support matrix

Outputs:

* campaign health
* promising regions
* exhausted regions
* current dominant failure mode

## 2. Knowledge Retrieval

Inputs:

* event family under consideration
* templates under consideration
* context/regime candidates
* trigger type candidates

Outputs:

* related mechanisms
* suitable templates
* plausible predecessor events
* adjacent families
* known pitfalls

This should come from the knowledge graph and curated docs.

## 3. Search Policy Selection

Choose one:

* `explore`
* `exploit`
* `narrow`
* `confirm`
* `pivot`
* `abandon_region`

Exactly one primary intent per run.

## 4. Experiment Proposal

Emit one registry-legal deterministic config.

This config should be machine-checked before execution.

## 5. Reflection Writer

After the run, produce structured memory:

* what was tested
* what was learned
* what is now promising
* what is now exhausted
* what should happen next

This reflection should be appended, not replace prior memory.

# Final memory design

The final memory system should be append-only and split into three levels.

## Level 1 — Immutable platform truth

* ledger
* artifacts
* support matrix
* registries

## Level 2 — Dynamic campaign summaries

* `campaign_summary.json`
* `search_frontier.json`
* `event_statistics.json`
* `campaign_findings.json`

## Level 3 — Agent reflection memory

Append-only structured reflections keyed by `program_id`.

This memory should never override the ledger. It only guides future decisions.

## Reflection schema

Each run should append:

```yaml
program_id: <program_id>
run_id: <run_id>
intent: explore|exploit|narrow|confirm|pivot|abandon_region

tested_region:
  trigger_types: [...]
  families: [...]
  events: [...]
  templates: [...]
  contexts: {...}
  directions: [...]
  horizons: [...]

outcome:
  evaluated_share: ...
  empty_sample_share: ...
  insufficient_sample_share: ...
  unsupported_share: ...
  skipped_share: ...

findings:
  promising_regions: [...]
  dead_zones: [...]
  structural_issues: [...]

next_policy:
  recommended_intent: ...
  recommended_focus: ...
  stop_condition: ...
```

That is the right memory shape.

# Final knowledge system design

The final knowledge system should be hybrid.

## Static curated knowledge

* event docs
* detector docs
* mechanism docs
* template docs
* regime docs
* heuristics docs

## Graph ontology

Nodes:

* Event
* Detector
* Template
* Regime
* MarketMechanism
* State
* Feature
* Campaign

Edges:

* `DETECTED_BY`
* `BELONGS_TO_FAMILY`
* `OBSERVES_MECHANISM`
* `PRECEDES`
* `SUITS_TEMPLATE`
* `MORE_LIKELY_IN`
* `PROMISING_IN`
* `FAILED_IN`
* `EXHAUSTED_IN`

## Dynamic campaign graph

Updated each run from:

* ledger
* summary
* frontier
* reflections

## Why this matters

This lets the agent reason like this:

* “`DEPTH_COLLAPSE` is a liquidity-fragility event”
* “it often precedes `VOL_SPIKE`”
* “continuation is theoretically suitable”
* “this campaign has not exhausted it”
* “the last two basis-family runs failed with empty samples”
* “therefore propose a bounded volatility-liquidity sequence confirmation run”

That is far better than naïve grid search.

# Final controller design

Implement one controller entrypoint.

## Recommended CLI

```bash
python -m project.pipelines.research.run_campaign_step --program_id <id> --agent_config <optional>
```

## Controller responsibilities

* check campaign state
* refresh intelligence
* gather knowledge context
* call LLM
* validate proposal
* execute if valid
* write artifacts
* update ledger
* update campaign graph
* append reflection
* emit step summary

## Output of one controller step

A step summary artifact:

```json
{
  "program_id": "...",
  "run_id": "...",
  "campaign_state_before": "active",
  "agent_intent": "narrow",
  "proposal_valid": true,
  "run_executed": true,
  "terminal_status_mix": {...},
  "campaign_state_after": "active",
  "next_recommended_action": "confirm"
}
```

This makes the loop easy to audit.

# Final campaign state machine

Add explicit campaign states.

## States

* `active`
* `halted_budget_exhausted`
* `halted_empty_sample`
* `halted_unsupported`
* `converged`
* `completed`
* `manually_paused`

## Transition rules

* budget exceeded → `halted_budget_exhausted`
* repeated high empty-sample runs → `halted_empty_sample`
* repeated unsupported-arity proposals → `halted_unsupported`
* no improvement over configured window → `converged`
* user stop or manual pause → `manually_paused`

This is cleaner than inferring everything ad hoc from ledger rows.

# Final agent prompt contract

The final agent should not receive the whole repo. It should receive a strict packet.

## Input packet

* campaign state
* `campaign_summary.json`
* `search_frontier.json`
* selected knowledge graph query outputs
* relevant registry excerpts
* support matrix excerpt
* last reflection memory records
* hard campaign limits

## Output packet

* one declared intent
* one concise rationale
* one deterministic `experiment_config.yaml`
* one next-step decision rule
* one post-run reflection template

That keeps the interface stable and machine-checkable.

# Final proposal quality rules

Before execution, the proposal should be rejected if it:

* violates registries
* violates support matrix
* exceeds campaign budget
* mostly targets exhausted regions
* mostly uses unsupported trigger structures
* duplicates the prior run with no meaningful change
* cannot state expected learning value

This should sit in the validation/controller boundary.

The project is now best positioned as:

That is the right final design because it preserves:

* statistical discipline
* semantic governance
* agent flexibility

while avoiding the two biggest failure modes of LLM systems in research platforms:

* uncontrolled semantic drift

The final agent for this project should be:

* **external**
* **stateful**
* **evidence-driven**
* **registry-bounded**
* **ledger-aware**
* **knowledge-augmented**
